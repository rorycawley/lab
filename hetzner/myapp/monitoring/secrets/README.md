# monitoring/secrets/

Sealed Secrets for the LGTM stack live here. Files matching
`*.sealedsecret.yaml` are generated by `bb monitoring-seal-secrets`,
encrypted with the cluster's Sealed Secrets controller public key, and
**safe to commit** — only the controller running in your cluster can
decrypt them.

## Files

- `grafana-admin-creds.sealedsecret.yaml` — Grafana admin user / password
- `minio-root-creds.sealedsecret.yaml` — MinIO root user / password (also
  consumed by Mimir as env vars for its S3 backend)

## Lifecycle

| Action | Command |
|--------|---------|
| First-time setup or rotation | `bb monitoring-seal-secrets` |
| Apply + install LGTM stack | `bb monitoring-install` |
| After rotation: pick up new values | `kubectl rollout restart -n monitoring deploy/grafana statefulset/minio-mimir deploy/mimir` |

## After a cluster rebuild

The Sealed Secrets controller's private key is cluster-scoped. When the
cluster is destroyed and recreated, the existing `*.sealedsecret.yaml`
files become unreadable to the new controller. Either:

1. Back up `kube-system/sealed-secrets-key*` Secrets before destroy and
   restore them after recreate, or
2. Re-run `bb monitoring-seal-secrets` to generate fresh credentials and
   overwrite the YAMLs (then commit and apply).
